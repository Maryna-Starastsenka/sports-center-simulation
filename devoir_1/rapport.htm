Hypothèses

Puisque le «Centre de données» et le «Répertoire de services» font partie du logiciel,
ils seront mentionnés, mais ne seront pas des acteurs dans les cas d'utilisation. Nous avons supposé qu'il faudrait quatre différents fichiers pour conserver les données.

  • Liste des membres : contient les informations personnnelles des membres.
  • Liste des professionnels : contient les informations personnelles des professionnels, leur domaine d'expertise et leurs séances.
  • Horaire des services : Contient les informations et l'horaire sur les services du jour.
  • Services fournis au gym : contient toutes les informations sur les différentes séances et services hebdomadaires offert au gym.

Dans ce rapport nous feront surtout mention du Centre de données et du Répertoire des services qui font la gestion des fichiers.
Puisqu'il est spécifié dans la description des besoins qu'un membre s'inscrit «pour le jour même», nous avons considérer qu'il faudrait
deux fichiers pour les services. Cela permettra par la même occasion d'avoir un horaire des séances de la journée même.

Il n'y a aucune information sur la façon dont les paiements et les abonnnements fonctionnent.
Il aurait pu être utile dans le cas d'un abonnement mensuel de suspendre l'abonnement automatiquement dans le logiciel s'il n'est pas renouvelé.
Cependant, dans le cas présent nous allons considérer que tous les membres seront suspendus manuellement.

Tâche 1 : Glossaire

Glossaire des termes clés
Code de service : Code unique utilisé pour référencer un service offert par un professionnel.
Membre : Client qui a accès illimité aux salles de sports et aux services d'un professionnel.
Numéro de membre : Numéro unique qui identifie un membre.
Numéro de professionnel : Numéro unique qui permet d'identifier un professionnel.
Professionnel : Expert en exercice physique ou en santé qui offre un service aux membres.
Séance : Service hebdomadaire offert par un professionnel pour les membres de #GYM.
Membre suspendu : Membre qui n'a plus accès au gym.
Capacité maximale : Quantité maximale de membres qui peuvent accéder à une séance.
Agent à la réception : Personne employée par le gym en charge d'interagir avec les clients et le logiciel.

Tâche 4 : Risques

    1. Risque sévère :
        - Fuite des données des utilisateurs.
        - Ressource humaine pour la production du livrable insuffisante (2 personnes pour un projet de 3 personnes).
    2. Risque important :
        - Une personne non-membre pourrait utilisé le code d'une personne membre pour s'autentifier sans abonnement.
    3. Risque éloigné :
        - L'interface du logiciel doit être assez intuitive pour être utilisée par un employé du gym qui n'est pas expert en informatique.
    4. Obstacle mineur :
        - La complexité du code doit être telle qu'une personne non-membre ne puisse en donner un au hasard et et tomber sur le code d'un vrai membre.

Tâche 5 : Exigences non-fonctionnelles

Identifions les besoins non-fonctionnels du système. 

    Utilisation : Le système doit avoir une interface convivial.

    Fiabilité :
    1. Le système doit être disponible 99.999% lors des heures d'ouverture du gym.
    2. Le système doit garantir la sécurité des données des membres et des professionnels.
    3. Le système peut perdre seulement l'information de la saisie courante (avant l'enregistrement).
    4. Le démarrage du système ne doit pas dépasser 10s.
    5. Le système doit être adaptable au besoins futurs du gym

    Performance : Le système doit répondre aux actions des utilisateurs en moins de 0.5s.

    Implémentation : Le système doit être testé sur une console Java.
