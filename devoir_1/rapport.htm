Tâche 1 : Glossaire

Glossaire des termes clés
Membre : Client qui a accès illimité aux salles de sports et aux servicces d'un professionnel.
Numéro de membre : Numéro unique qui identifie un membre.
Professionnel : Expert en exercice physqiue ou en santé qui offre un service aux membres.
Numéro de professionnel : Numéro unique qui permet d'identifier un professionnel.
Séance : Service hebdomadaire offert par un professionnel pour les membres de #GYM.
Code de service : Code unique utilisé pour référencer un service offert par un professionnel.

En fait c'est peut-être plus les termes spécifiques à un gym qu'il faut définir tel que ( expert en exercice physique (yoga, cross-fit, etc.), expert de la santé physique (nutritionniste, physiothérapeute, massothérapeute), etc. ??)

Tâche 3 : Cas d'utilisation

    Cas d'utilisation : Inscription au gym
    But : Le client s'inscrit pour devenir membre du gym.
    Préconditions : Aucune
    Acteurs : Nouveau client, Agent à la réception (secondaire)
    Scénario principal :
    1. Le logiciel affiche le menu.
    2. L'agent choisit Inscription d'un nouveau membre dans le logiciel.
    3. Le logiciel affiche le formulaire d'inscription.
    4. L'agent entre les informations que le client lui fournit.
    5. L'agent appuit sur enregistrer.
    6. Le logiciel envoie les données au Centre de données.
    7. Le logiciel fournit un numéro unique au nouveau membre.
    Scénarios alternatifs :
    Postconditions :Le client a maintenant un numéro de membre unique qui lui donne accès au gym. 
    Cas d'utilisation : Demande d'accès au gym
    But : Le client se présente à la réception et demande l'accès au gym.
    Préconditions : Le client doit être membre
    Acteurs : Le membre (principal), Agent à la réception(secondaire)
    Scénario principal :
    1. Le logiciel affiche le menu
    2. L'agent choisit Validation du numéro de membre.
    3. L'agent entre le numéro de validation dans le logiciel.
      3.1. Le logiciel vérifie le numéro.
      3.2. Le logiciel vérifie que le numéro est associé à un abonnement valide
      3.3. Le logiciel affiche «Numéro valide».
    4. L'agent confirme et permet l'accès au membre.
    Scénarios alternatifs : ramifications du déroulement principal, alternatives possible à certaines étapes
    3.2.a Le logiciel ne trouve pas le numéro et renvoie «Numéro invalide».
    3.3.a Le cas d'utilisation retourne à l'étape 3.
    3.2.b Le logiciel ne voit pas d'abonnement valide.
    3.3.b Le logiciel affiche «Membre suspendu».

Tâche 4 : Risques

1. Risque sévère - Fuite des données des utilisateurs - Ressource humaine pour la production du livrable insuffisante (2 personnes pour un projet de 3 personnes). 2. Risque important - Une personne non-membre pourrait utilisé le code d'une personne membre pour s'autentifier sans abonnement. 3. Risque éloigné - L'interface du logiciel doit être assez instinctive pour être utilisé par un employé du gym qui n'est pas expert en informatique. 4. Obstacle mineur La complexité du code doit être tel qu'une personne non-membre ne puisse en donner un au hasard et et tomber sur le code d'un vrai membre.

Tâche 5 : Exigences non-fonctionnelles

Identifions les besoins non-fonctionnels du système. 

    Fiabilité :
    1. Le système doit être disponible 99.999% lors des heures d'ouverture du gym.
    2. Le système doit garantir la sécurité des données des membres et des professionnels.
    3. Le système peut perdre seulement l'information de la saisie courante (avant l'enregistrement).
    4. Le démarrage du système ne doit pas dépasser 10s.
    5. Le système doit être adaptable au besoins futurs du gym

    Performance : Le système doit répondre aux actions des utilisateurs en moins de 0.5s.

    Implémentation : Le système doit être testé sur une console Java.
